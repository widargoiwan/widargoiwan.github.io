Using keys in databases

September 15, 2019

Having gone through the bulk of SQL without a theoretical framework, other than ER diagrams, I believe that it is time to consider the concept of keys in databases. Keys are fundamental to relations: they are supposed to uniquely identify each tuple in a relation.

There are three types of keys that are important:

  1. A superkey is a set of (one or more) attributes that uniquely identifies a tuple in the relation. A superkey may contain extraneous attributes. The set of all attributes uniquely identifies each tuple, and it is considered a superkey.
  2. A candidate key is a set of (one or more) attributes that uniquely identifies a tuple in the relation, for which no proper subset is a superkey. In other words, the set for a candidate key is a superkey from which all extraneous attributes have been removed.
  3. A primary key is a candidate key that is chosen arbitrarily by the database designer as the principal means of identifying tuples in a relation.

A key, whether as a primary, candidate or super key, is a property of the entire relation, rather than of the individual tuples. Any tuple is prohibited from having the same value on the key attributes at the same time.

Candidate Keys

There are also a couple more things to note for candidate keys:

  1. A candidate key can never be null or empty.
  2. There can be more than one candidate key for each table.

The use of keys naturally leads to the setting of constraints on the relation. This allows us to design a database that is logically related.
